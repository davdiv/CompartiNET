## Development Guidelines

### Scripting Language

When writing ad-hoc scripts to inspect system state or parse JSON, never use Python, use Node.js (`node -e ...`) instead. This keeps the tooling consistent with the rest of the repository.

### Testing Commands

When you are unsure about the behavior or output format of system commands (`ip`, `wg`, `bridge`, etc.), reproduce the scenario and run the commands yourself in the test container (cf `./tests/runInContainer.sh`). If you cannot reproduce the scenario you need to test, ask the user. Don't call docker directly and don't run commands outside the container, call `./tests/runInContainer.sh <command-line>`.

### Namespace Execution

Always use `getNetnsPrefix` (`src/common/model/commands.ts`) when constructing commands for a specific network namespace. For a non-empty namespace name it prepends `ip netns exec <netns>`, which properly mounts `/sys` in the namespace; for the default namespace (empty string `""`) it returns an empty array (no prefix).

### Plain Object Map Safety

When using plain JavaScript objects as dictionaries with arbitrary or user-controlled keys, always guard key accesses with `Object.hasOwn` (or equivalent) to avoid prototype-pollution bugs and collisions with inherited properties such as `hasOwnProperty`, `constructor`, or `toString`.

For example, prefer:

```typescript
if (Object.hasOwn(obj, key)) {
  const value = obj[key];
  // ...
}
```

over unguarded `obj[key]` when `key` is not a known, literal string.

### undefined / null

Null and undefined should always be processed the same way. Don't use `undefined` when comparing, use `null` (`== null` should be used instead of `=== null` or `=== undefined`). When producing values, use undefined.

## Architecture Overview

CompartiNET is a modular network manager for Linux that uses a "Desired State" vs "Current State" reconciliation pattern.

### 1. Network Model & Features

The system represents the network as a `NetworkModel` (Namespaces -> Interfaces -> Routes). The default namespace is represented by an empty string key `""`.

- **Desired State**: Generated from a list of `Feature` objects — a union of `NetworkCreateAction` (direct network operations such as `CreateNamespace`, `CreateBridge`, `AddIpAddress`, `SetWireguardConfig`, etc.), expandable features (`ConfigDirectory`, `ConfigFile`, `DhcpClient`), and `ServiceSpec`. `createFeatureSystem` (`src/common/features/index.ts`) recursively expands expandable features via registered handler functions (in `src/node/features/`) until only `NetworkCreateAction` and `ServiceSpec` remain, then builds the desired model with `createDesiredModelFromBasicFeatures`. See `sampleConfig.yaml` for a reference of all supported action types.
- **Current State**: Generated by polling the system using `iproute2` (via `src/node/collectState.ts`) and parsing the JSON output.

### 2. Modular Reconciler

The Reconciler (`src/common/reconcile/`) computes the difference between Current and Desired states.

- **Action Model**: It generates abstract `Action` objects (e.g., `CreateNamespace`, `AddIpAddress`, `AddBridge`, `AddVeth`, `SetWireguardConfig`).
  - Every action has an `applyX` counterpart that mutates a `NetworkModel` to simulate the action's effect.
  - **Action/Command Parity Rule**: An `applyX` function must only succeed when the corresponding real command would also succeed. If a command is idempotent and succeeds on duplicate application, the `applyX` may silently accept the duplicate; otherwise it must throw an error matching the command's failure mode. This parity increases confidence that if the action succeeds, the real command will succeed as well. Therefore, all validation checks performed by the real command should also be performed by the action, allowing issues to be caught before they could have more severe consequences on a live system.
  - The reconciler uses this to build an `expectedModel` as it plans changes, and integration tests use it to verify that applying an action to the current state produces the expected result.
- **Command Generation**: Each action file in `src/common/model/actions/` exports a `commandForX` function that converts the action into executable `iproute2`/`wg` command arrays. `commandForActions()` in `src/common/model/actions/index.ts` aggregates them into the full command list.
- **ReconcileContext** (`src/common/reconcile/context.ts`): Tracks the generated actions, errors, and a `currentModel` that starts as the current state and is progressively updated via `context.apply(action)` (which internally calls `applyAction`) as the reconciler plans changes.

### 3. Reactive Computation (signalium)

The `compartinet-manager` service is built on [signalium](https://github.com/Signalium/signalium) (a reactive signals library). The entire reconciliation pipeline is expressed as a graph of reactive computations that re-run automatically whenever their inputs change.

Key primitives used:

- `reactive(fn)` — creates a lazy, cached async computation. Re-evaluates only when a dependency (another `reactive` or `signal`) has changed.
- `signal(value)` — a mutable reactive cell; writing to it invalidates all downstream computations.
- `notifier()` — a manual invalidation trigger (used by `refreshState` in `reconciler.ts` to re-poll current system state after applying commands).
- `watcher(fn)` — a side-effecting observer that re-runs `fn` whenever dependencies change; drives the top-level reconciliation loop in `reconciler.ts`.
- `getContext(key)` / `withContexts(...)` — signalium's context API, used to pass the `MarkReloadable` symbol into feature handlers so that config-file reads register themselves as reactive dependencies (enabling automatic reload when files change).

The reactive graph in `reconciler.ts`: `currentState` → `desiredState` → `reconciliationResults` → `applyReconciliation`, all wrapped in a `watcher`. Any change to config files or system state automatically propagates through the graph and triggers a new reconciliation pass.

Note: `currentState` (system state collection) is currently also refreshed on a fixed 30-second `setInterval` in `compartinet-manager/main.ts`. Continuous watching of system state (to avoid this timer) is a future goal but not yet implemented.

**Feature handlers are pure reactive functions** — they must not perform side effects directly. Instead, a feature handler that needs a background service (e.g. a DHCP client) returns a `ServiceSpec` object alongside its `NetworkCreateAction`s. The service manager runs the service and exposes its output via `ctx.getServiceOutput(serviceKey)`, which is backed by a reactive `signal`. This means feature handlers can call `ctx.getServiceOutput` to reactively read service output (e.g. a DHCP lease) and derive the appropriate network actions from it — when the service output changes, the feature re-evaluates automatically and the reconciler applies the updated desired state. The `DhcpClient` feature (`src/node/features/dhcpClient.ts`) is the canonical example of this pattern.

### 4. Node.js Service & CLI Tools

The backend code (in `src/node/`) provides:

- **State collection**: `collectState.ts` polls the system to build the current network model.
- **Command execution**: `spawnUtils.ts` runs `iproute2` and `wg` commands. It supports a special `CommandArg` types beyond plain strings: `TempFileArg` creates a temporary file with `0o600` permissions to pass secrets (e.g., WireGuard keys) without exposing them in process lists and `DefaultNetnsArg` allows to add the path to the current network namespace (`/proc/${process.pid}/ns/net` which is different than `/proc/self/ns/net` when passed to a child process spawned in a different namespace)
- **CLI entry points** in `src/node/cli/`:
  - `compartinet` -- main CLI
  - `compartinet-manager` -- systemd service
  - `netns-worker` -- child process spawned to open sockets inside a network namespace

## Project Structure

- `src/common/features/`: Generic feature expansion framework (`createFeatureSystem`, `createFeaturesFactory`). The actual feature handlers live in `src/node/features/`.
- `src/node/features/`: Node-specific feature handlers (`ConfigDirectory`, `ConfigFile`, `DhcpClient`) that expand high-level config entries into `NetworkCreateAction` and `ServiceSpec` objects.
- `src/common/model/`: Core network model definitions, action abstractions, and current-state generators.
- `src/common/reconcile/`: The reconciliation engine (namespaces, hardware, interfaces, routes).
- `src/node/`: System-level implementation (state collection, command execution, CLI tools).
- `tests/unit/`: Unit tests for model generation and reconciler logic.
- `tests/integration/`: Integration tests that run real `iproute2` and `wg` commands inside a privileged docker container (built from `tests/container/Dockerfile`). The harness (`tests/integration/harness.ts`) provides `runActionAndVerify`, which applies a single action and asserts that the collected after-state matches the simulated expected state.
